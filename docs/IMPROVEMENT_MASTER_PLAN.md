# mermark 개선 마스터플랜 (재귀 수렴 → 최단경로)

> `recursive-improvement-plan` 산출물. 북극성 단일축, 차수 순차 수렴, 누적안 최단경로 종합. **계획 문서 — 구현은 별도.**
> 기준 버전: v0.4.0 · 작성: 2026-06-24 · **개정 2판**(북극성 정정: 볼트 패리티 → 경량 단일파일 + 플러그인)

---

## 북극성 (정정판)

**볼트의 무게 없이 단일 마크다운 파일을 CLI로 즉시 열어, Obsidian급 *편집·렌더 품질*로 보고 쓰는 가벼운 에디터. 그 위에 Obsidian식 *플러그인 시스템*으로 편집·렌더 능력을 코어 비대화 없이 확장한다.**

핵심 정체성: **반(反)볼트.** 볼트 여는 불편함을 없애려 만든 앱이다 — 파일트리·탭·볼트검색·그래프·백링크 등 "노트 앱" 기능은 **북극성 위배**(그건 Obsidian이 함). "Obsidian급"은 **편집 품질 + 플러그인 확장성**이지 워크스페이스가 아니다.

### 성공 기준 (평가축)
1. **즉시성·경량** — CLI로 단일파일 즉시 열림, 빠른 콜드로드, 작은 번들, 외부 의존 0. *vs-Obsidian 최대 차별점.*
2. **편집·렌더 품질(Obsidian급)** — 라이브프리뷰·Mermaid·위키링크·콜아웃·풋노트·임베드·검색·아웃라인이 매끄럽고 충실.
3. **확장성(플러그인)** — 서드파티/사용자가 편집·렌더·커맨드·설정을 플러그인으로 추가, 코어 무변경.
4. **신뢰성** — atomic save·conflict guard·안 깨지는 렌더.

### 현재 상태 (v0.4.0)
라이브프리뷰·Mermaid·KaTeX·코드·표·콜아웃(13종)·풋노트(양방향·hover)·위키링크·이미지/비디오/유튜브 임베드·하이라이트·fold·edit/read·Vim·줌·세션영속·paste-to-link·설정 SSOT(테마 비주얼 에디터)·번들 CLI·atomic save·conflict guard. **내부** feature 레지스트리(InlineFeature/BlockFeature, `bd403aa`)는 있으나 **외부 플러그인 API·로딩은 없음**. 콜드로드 시간 미계측. 문서 내 검색 없음.

## 의존 그래프 (DAG)

```
[P1 플러그인 시스템] ──┬─→ [P10 플러그인 설정 API]   ← [설정 SSOT 기존]
  (최대 enabler)       ├─→ [P11 로딩·발견]
                       ├─→ [P12 코어 dogfood]
                       ├─→ [P13 권한·샌드박스]
                       ├─→ [P14 API 버저닝·문서]
                       └─→ [P15 테마 플러그인]      ← [theme-as-JSON 기존]
[P5 팔레트] ──→ (플러그인 커맨드 등록 표면)  ← P1과 시너지
[P3 검색] [P4 아웃라인] [P6 임베드크기] [P8 frontmatter]  (독립, 무의존)
[P2 콜드로드] [P9 열기마찰] [P7 export]  (독립)
```

핵심 긴장: **P1(플러그인)이 최대 enabler지만 L·아키텍처.** P3·P4·P6·P8(편집 코어)은 무의존·고임팩트·즉시 체감. → 코어 편집 표면을 먼저 만들어 **플러그인 확장점이 뭘 노출해야 하는지 학습한 뒤** P1을 설계해야 잘못된 추상화를 피한다(discover-then-abstract).

## 마스터플랜 (최단경로)

### M1 — 단일문서 편집 품질 (무의존·고임팩트·빠른 체감)
**P3 검색·치환 · P4 아웃라인 네비 · P6 임베드 크기 · P8 frontmatter · P5 커맨드 팔레트**
- 왜 먼저: 플러그인 대공사 없이 "편집 품질(기준 2)"을 즉시 끌어올림. 검색 부재가 현재 최대 마찰. **부수효과: 여기서 만든 기능이 P1 플러그인 확장점의 레퍼런스가 됨**(어떤 훅이 필요한지 학습).

### M2 — 즉시성·열기·내보내기 (차별점 lock)
**P2 콜드로드 예산·계측·최적화 · P9 열기 마찰 제거 · P7 단일문서 export**
- 왜: "즉시 연다(기준 1)"가 vs-Obsidian 핵심. 계측 없이 최적화 없다 — 예산을 박는다. 열고(파일연결·$EDITOR) 내보내는(PDF/HTML/Mermaid) 단일파일 라이프사이클 완성. 대부분 독립.

### M3 — 플러그인 시스템 (확장 기반)
**P1 공개 API + P11 로딩 · P13 권한·샌드박스 · P12 코어 dogfood**
- 왜 여기: M1·M2가 코어를 안정화하고 **확장점 표면을 드러낸 뒤** 설계. 기존 내부 레지스트리(`bd403aa`)를 **공개·로드 가능·샌드박스된 API**로 승격. 코어 일부(예: 콜아웃/임베드)를 플러그인으로 dogfood해 API를 검증. 기준 3의 본체. **경량 유지가 제약** — 플러그인 로딩이 콜드로드(기준 1)를 깨면 북극성 위배.

### M4 — 플러그인 생태계 (깊이)
**P10 플러그인 설정 UI · P14 API 버저닝·문서 · P15 테마 플러그인**
- 왜 마지막: P1 위에서 생태계를 키움. 플러그인이 설정을 노출(SSOT 위), API를 버전·문서화해 서드파티가 안전하게 붙고, 테마를 배포 가능하게. "Obsidian식 플러그인" 꿈의 완성.

### 곁가지 (제외 — 북극성 위배 or 무게 위반)
- **볼트·멀티노트·파일트리·탭·볼트검색·⌘O·백링크·태그·그래프·템플릿·분할패널·연동패널** — 전부 "노트 앱" 기능, **반볼트 정체성 정면 위배**. Obsidian이 하는 영역. 제외.
- **클라우드 동기화** — "로컬·외부서비스 0" 위배. 제외.
- **플러그인 마켓플레이스(서버·발견 UI)** — 무거움. 경량 로딩(폴더/파일/URL)으로 충분. 코어 생태계 성숙 후에만 검토.
- **모바일** — 데스크톱 CLI 정체성 밖. 제외.

## 다음 단계
**계획 문서**다 — 구현은 `mermark-dev`로 마일스톤·항목 단위 위임. 권장 착수: **M1의 P3 검색**(무의존·최대 마찰 해소) → M1 나머지 → M2 → P1 플러그인. 북극성/축을 또 바꾸면 이 문서를 재귀 루프로 다시 돌린다.

> 사용 모델 메모: P1 플러그인을 감정적 1순위로 당기고 싶을 수 있으나, 최단경로 논리는 "코어 편집 표면을 먼저 만들어 확장점을 학습 → 그 뒤 API 추상화"다(M1→M3). 플러그인을 더 일찍 당기면 잘못된 추상화 리스크. 결정은 사용자 몫.
